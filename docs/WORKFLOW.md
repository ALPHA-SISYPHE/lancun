# 固定工作流程

## 文档状态

- 版本：0.2
- 状态：讨论中

## 总流程

```text
作业要求
→ 产品需求 Product
→ 设计 Brief Constraints
→ 图片/视觉方向稿
→ 用户选择主方向 Human Select
→ 提取 Tokens & Specs
→ Codex 分阶段实现
→ Playwright 截图对比
→ 视觉修复 Repair Loop
→ 功能、响应式与构建测试
→ 实验报告与最终提交
```

## 阶段门槛

### 阶段 A：需求锁定

产出：`PROJECT_BRIEF.md`、`PAGE_STRUCTURE.md` 初版。

进入下一阶段前必须确认：主题、受众、核心体验、页面范围、必做功能。

### 阶段 B：视觉方向探索

产出：`DESIGN_BRIEF.md` 和 2～3 个可比较的方向稿，可以由图像生成、参考图整理或低保真框架组成。

由用户亲自选择方向；不得让 AI 默认替用户完成最终选择。排版和动效也需要逐步征询用户意见。

### 阶段 C：设计系统锁定

产出：`DESIGN_SYSTEM.md`，包含色值、字号、间距、圆角、阴影、组件状态和动效规格。

确认颜色、字体、间距、组件、响应式和动画规则后再开发正式页面。

### 阶段 D：分页面实现

先制作结构清晰、视觉未完全精修的框架页面，再按用户确认的方向逐步填充和优化。建议顺序：全站基础 → 海洋美景首页 → 互动地图/科普 → 数据与污染现状 → 生物档案 → 行动打卡与表单。

每个页面都执行：框架实现 → 用户确认结构 → 视觉填充 → 浏览器检查 → 截图对比 → 修复 → 验收。

### 阶段 E：整体验收

按 `ACCEPTANCE.md` 检查功能、内容、视觉、性能、引用和作业材料。

### 阶段 E+：自动化检查（可选增强）

- 使用 Playwright 打开主要页面并生成桌面端、移动端截图。
- 对比设计方向稿或已确认的视觉规范，记录明显偏差。
- 执行链接检查、控制台错误检查和构建检查。
- 自动化结果只作为辅助证据，最终视觉判断仍由用户确认。

### 阶段 F：沉淀定制 Skill

至少完成一个代表性页面和一次修复循环后，再把真正重复、有效的流程整理为 Skill。Skill 应封装流程，不应替代用户对视觉方向的选择。

## 决策记录规则

- 确认的决定立即写入相关文档。
- 被替代的决定应注明替代原因，不保留相互冲突的现行规则。
- `TASKS.md` 只记录当前状态；长期规则写入 `docs/`。
