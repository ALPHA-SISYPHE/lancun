## 1. 页面定位

中文名：**我的海洋守护档案**  
英文名：**Ocean Guardian Profile Center**

本页是用户在澜存网站中的个人行动记录与本地档案中心：以「我是怎样守护海洋」为叙事主线，把行动中心产生的打卡、勋章、志愿报名和公益支持沉淀为一份可回看的守护档案。

它应该体现：

1. 守护者身份与个人资料。
2. 已完成的海洋行动和持续打卡进度。
3. 已获得的勋章与下一枚勋章目标。
4. 已报名的志愿活动。
5. 已支持的公益项目。
6. 显示、动效等个人偏好。
7. 数据仅保存在当前浏览器的边界与管理入口。

它不是：

- 普通后台设置页。
- SaaS 或管理系统 Dashboard。
- 大面积纯白卡片拼接的 AI 页面。
- 与全站深海纪录片叙事割裂的账户详情页。

---

## 2. 本次页面范围与不可触碰边界

本规则只约束用户中心页面及其直接服务于档案展示的专用样式、渲染逻辑。

**不得修改：**

- 首页。
- 我们的海洋页。
- 海在呼救页。
- 生物档案页。
- 行动中心页。
- 全站导航结构。
- 右上角头像入口、账户菜单和登录/注册流程。
- 其他页面业务逻辑。

头像入口可以保持既有逻辑；用户中心仅承接其跳转后的档案展示。行动中心仍是打卡、报名与公益支持的发生地，用户中心只读取、汇总、解释和管理本地记录，不重复创建相互冲突的数据源。

## 3. 视觉气质

### 3.1 应有感受

- 深海背景视频之上的半透明档案舱。
- 海洋纪录片式冷静、沉浸、克制，而非办公后台。
- 守护身份、行动轨迹、荣誉反馈与本地优先隐私感并存。
- 信息密度充足但节奏清晰；每个模块有目的，不用空白硬撑高度。
- National Geographic 式自然叙事与 Apple Environmental Report 式高级排版。

### 3.2 绝对禁止

1. 改成普通后台 Dashboard。
2. 使用大面积纯白实心面板。
3. 使用紫蓝渐变 AI 风。
4. 使用强阴影、新拟态或厚重卡片堆叠。
5. 使用默认浏览器表单和按钮样式。
6. 为了“高级感”制造大面积无内容空白。
7. 侧栏、菜单和内容区错位。
8. 为凑版面无意义撑高功能区。
9. 出现横向滚动、文字遮挡、图片拉伸或错误层叠。
10. 破坏其他页面功能或改写行动中心的 localStorage schema。

- 主档案面板不得跨满整屏，也不得紧贴视口边缘。

- 圆角、阴影、动效时间服从 `DESIGN.md`；hover 仅允许克制的透明度或 `translateY(-2px)` 反馈。
- 所有持续或装饰性动效必须服从 `prefers-reduced-motion` 与用户的减动效偏好。

## 5. 信息架构（后续实现目标）

用户登录后，页面按以下结构组织；可使用侧边档案导航或同等层级清晰的布局，但不能回退为普通设置菜单。

1. **守护者档案首屏**
   - 头像、显示昵称、守护身份、加入日期。
   - 一句简短的守护状态说明。
   - 累计行动、打卡天数、勋章、志愿报名、公益支持等紧凑概览。
2. **行动足迹**
   - 以时间和类型整理打卡、志愿报名、公益支持等记录。
   - 记录应能回链至行动中心相应模块，但不复制行动中心完整表单。
3. **打卡与勋章**
   - 当前连续打卡、最长连续天数、当月进度、已解锁勋章与下一目标。
   - 勋章使用铜、银、金的克制分级，不制造夸张游戏化效果。
4. **志愿活动**
   - 展示当前用户的报名记录、活动时间、地点、状态和提交时间。
5. **公益支持**
   - 展示当前用户的支持记录、项目、金额、时间与模拟说明。
6. **资料与偏好**
   - 用户资料摘要。
   - 视频背景、减少动效等既有偏好设置；表单必须具备标签、校验、状态反馈和清晰焦点。
7. **本地数据说明**
   - 明确说明本页不连接服务器，数据只保存在当前浏览器。
   - 提供退出登录与清除数据的风险说明；破坏性操作必须二次确认。

### 5.1 高级空状态

缺少记录时，不能留一片空白，也不能用“暂无数据”草率结束。每个空状态应包含：

- 与区块语义一致的短标题与一句解释。
- 一个低干扰的线性图标或微型档案标记，不能使用抢眼插画。
- 指向行动中心的单一、清晰 CTA。
- 紧凑高度和半透明面板，仍保持完整信息层级。

未登录时应显示守护档案入口的引导状态，并复用既有头像菜单登录逻辑；不得在本页另造登录流程。

## 6. localStorage 数据契约

### 6.1 读取原则

- 用户中心是**只读汇总端**；数据创建、修改和删除的主逻辑仍归行动中心或既有账户模块。
- 读取失败、JSON 格式损坏或字段缺失时，须降级为空数组/空对象并显示可理解的空状态，不能使页面报错。
- 所有按用户区分的数据以当前登录用户的 `username` 精确匹配，不得把其他本机用户的记录展示到当前档案。
- 不连接服务器、不伪造实时状态、不接入真实支付或真实志愿服务。

### 6.2 行动中心已落地 schema（当前权威）

| 业务 | 实际 key | 读取与筛选规则 |
|---|---|---|
| 打卡记录 | `ocean-action-checkins.{username}` | 数组；每项至少包含 `date`、`actionType`、`createdAt`，用于累计天数、当月进度、连续打卡与行动足迹。 |
| 勋章 | `ocean-action-badges.{username}` | 对象；以 `unlockedIds[]`、`unlockedAt{}` 为准，并与行动中心已定义的勋章目录关联展示。 |
| 志愿报名 | `ocean-action-volunteer-registrations` | 全局数组；仅展示 `item.username === currentUsername` 的记录。 |
| 公益支持 | `ocean-action-donations` | 全局数组；仅展示 `item.username === currentUsername` 的记录。 |

> 用户需求中列出的 `ocean-action-checkins`、`ocean-action-badges` 是这两类 key 的逻辑前缀。实现时必须补上 `.{username}`，不得错误读取无用户名的共享 key。

### 6.3 用户资料与偏好

| 业务 | 目标 key | 用途 |
|---|---|---|
| 用户档案 | `ocean-user-profile` | 可编辑的显示昵称、守护身份、头像偏好等非敏感档案字段。 |
| 用户偏好 | `ocean-user-preferences` | 视频背景、减少动效等展示偏好。 |

当前账户模块以 `ocean-auth-users` 和 `ocean-auth-current-user` 保存登录账户；后续实现上述两个目标 key 时，必须以当前账户为身份来源，并保留兼容回退，不得复制或存储密码。

兼容顺序固定为：

1. 读取 `ocean-user-profile` / `ocean-user-preferences` 中当前用户对应内容。
2. 缺失时，从 `ocean-auth-current-user` 与已登录账户的安全字段（昵称、身份偏好、邮箱、创建时间）补齐展示。
3. 仅为兼容旧档案展示时读取 `lancun.account`、`lancun.session`；不得继续扩展这些旧 key。

任何对资料或偏好的写入都必须按当前用户名隔离，避免多账户在同一浏览器中互相覆盖。

## 7. 响应式、无障碍与质量门槛

必须覆盖至少三个宽度：

| 宽度 | 规则 |
|---|---|
| 1180px 及以上 | 档案导航与主内容稳定对齐；统计和记录保持紧凑，不拉伸成空洞大块。 |
| 768px–1179px | 允许导航收窄或转为横向标签；内容改为合理双列/单列，不牺牲文字可读性。 |
| 375px–767px | 单列优先；操作区触摸目标不小于 44×44px；不得出现横向滚动。 |

同时必须满足：

- 导航、按钮、标签、表单和破坏性操作均可用键盘操作。
- 焦点状态清晰；动态统计和提交结果使用合适的 `aria-live`。
- 图片使用正确的 `object-fit` 与尺寸约束，禁止拉伸。
- 表格在窄屏应转为可读的记录列表或可横向滚动的局部容器；页面本身不得横向溢出。
- 背景视频加载失败时，保留可读的深海静态降级背景。

## 8. 后续实施与验收规则

1. 每次仅完成用户明确指定的一个阶段，不顺手推进下一阶段。
2. 每次修改前先读取本文档、[`ACCOUNT_SYSTEM_RULES.md`](ACCOUNT_SYSTEM_RULES.md)、[`DESIGN.md`](../DESIGN.md) 与行动数据契约。
3. 涉及页面视觉或 CSS 时，额外读取 `.cursor/skills/lancun-design/SKILL.md`。
4. 每一阶段至少检查桌面端与移动端，验证登录/未登录、空状态、有记录状态、数据损坏降级、导航、链接与控制台错误。
5. 页面和布局变更后，按实际 DOM 更新或新增针对用户中心的 smoke 验证；验收必须覆盖当前用户筛选和无横向溢出。
6. 完成后停止，并明确列出本阶段改动的文件；未经用户确认不得把后续视觉探索当作最终实现。
